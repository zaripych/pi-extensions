# pi-review implementation plan

## Goal

Build `pi-review`, a pi extension that provides a `/review` command for code review. It spawns an isolated `pi` subprocess as a reviewer with a fresh context window, read-only-by-default code exploration tools, a restricted git-inspection tool, and a structured review rubric. Findings are injected back into the parent conversation for the user and main agent to act on.

The algorithm closely follows the review implementation in `codex-rs` (documented in `review-investigation.md`), adapted from Rust/native sub-agent to pi's extension + subprocess model.

## Non-goals

- Interactive finding selection UI. Findings are displayed; the user decides what to do.
- Project-local config. Single global config only.
- Merging multiple config sources.
- Real-time streaming of reviewer subprocess progress.
- Giving the reviewer arbitrary write access or unrestricted shell access by default.
- Treating reviewer isolation as a security boundary. The lockdown is operational: avoid accidental mutation, keep the reviewer focused, and reduce context leakage from the parent agent. It is not intended to be a sandbox for hostile repositories, malicious local files, untrusted extensions, or multi-user adversarial scenarios.

## Resolved decisions

### Execution model

Spawn a `pi` subprocess in JSON mode (`--mode json -p --no-session`), same pattern as the subagent example extension. The reviewer needs tools to do a real review — read files, inspect diffs, inspect context around changes, and search the codebase. A single LLM call with a pasted diff is insufficient.

The subprocess is isolated from the parent conversation and runs with deterministic prompt/resource loading:

- no session persistence (`--no-session`)
- JSON event stream for machine parsing (`--mode json`)
- print/non-interactive mode (`-p`)
- reviewer rubric replaces the default coding-agent prompt (`--system-prompt <system-prompt-path>`), not appended to it
- auto-discovered extensions disabled (`--no-extensions`)
- context files disabled (`--no-context-files`)
- skill discovery disabled (`--no-skills`)
- prompt template discovery disabled (`--no-prompt-templates`)
- only explicitly configured extensions are loaded with `-e`, with the `pi-review` companion extension loaded first so internal tools win name conflicts

The subprocess is spawned using the same `getPiInvocation()` pattern as the subagent example extension (resolves the `pi` binary path), but without sync filesystem APIs. The spawner captures JSON mode output line by line, collects parsed JSON events plus stderr and exit code, and returns them to the command handler.

### Reviewer isolation model

The default reviewer subprocess is read-only from the model's perspective. This is not a security sandbox. The purpose is to avoid inadvertent file mutation, reduce parent-agent context leakage, and give the reviewer a focused tool surface for code review.

User-configurable tools are exactly `config.tools`. The config file is the source of truth for user-selected reviewer tools; implementation code must not append user-facing default tools behind the user's back.

Internal companion tools are implementation details and are not listed in config:

- `reviewer-git` — restricted git inspection for stock review targets
- `finish-review` — terminating structured-output submission

These internal tools are always added by loading the companion extension and appending them to the subprocess `--tools` allowlist.

Default config tools:

- built-in read/search/list tools: `read`, `grep`, `find`, `ls`

Default subprocess command shape:

```bash
pi --mode json -p --no-session \
  --system-prompt <system-prompt-path> \
  --no-context-files \
  --no-skills \
  --no-prompt-templates \
  --no-extensions \
  -e <review-tools-extension-path> \
  --tools read,grep,find,ls,reviewer-git,finish-review \
  --model <model> \
  "<task prompt>"
```

With configured extensions, users add both the extension path and its tool names to the same config. Internal tools still stay out of config:

```yaml
extensions:
  - ${getAgentDir()}/extensions/my-review-helper.ts

tools:
  - read
  - grep
  - find
  - ls
  - my_review_helper
```

The subprocess command then uses `config.tools` plus internal tools:

```bash
pi --mode json -p --no-session \
  --system-prompt <system-prompt-path> \
  --no-context-files \
  --no-skills \
  --no-prompt-templates \
  --no-extensions \
  -e <review-tools-extension-path> \
  -e ${getAgentDir()}/extensions/my-review-helper.ts \
  --tools read,grep,find,ls,my_review_helper,reviewer-git,finish-review \
  --model <model> \
  "<task prompt>"
```

Important constraints:

- `extensions` in YAML are supported and loaded explicitly with `-e` while auto-discovery remains disabled by `--no-extensions`.
- The `pi-review` companion extension is loaded **before** configured extensions. Pi keeps the first extension tool registration for duplicate names, so this guarantees internal `reviewer-git` and `finish-review` definitions win if a configured extension uses the same names.
- Tools registered by configured extensions are not active unless their names are listed in `tools`; pi's `--tools` allowlist applies to built-in, extension, and custom tools.
- `finish-review` and `reviewer-git` are internal implementation details and must not be configured by users.
- The subprocess arg builder appends `reviewer-git` and `finish-review` to the final `--tools` value after `config.tools`.
- The subprocess arg builder uses `--system-prompt`, not `--append-system-prompt`, and includes `--no-context-files`, `--no-skills`, and `--no-prompt-templates` so project context, skills, and prompt templates do not leak into the reviewer prompt.
- `--dangerously-skip-permissions` is not a pi flag and must not be used.
- Disabling `write` and `edit` is not enough if `bash` is enabled, because `bash` can mutate files. Therefore `bash` is not enabled in the default config.
- Configurable `extensions` and `tools` are explicit isolation overrides. Extension paths and non-default tool names are documented as weakening the default read-only/focused review environment, and covered by tests so the generated default config remains mutation-free.

### Restricted git inspection tool

The companion extension registers `reviewer-git`, a read-only git inspection tool. It exposes explicit actions instead of raw shell commands.

Allowed actions:

| Action             | Git command                                                       | Purpose                                      |
| ------------------ | ----------------------------------------------------------------- | -------------------------------------------- |
| `statusShort`      | `git status --short`                                              | Detect staged, unstaged, and untracked files |
| `diff`             | `git diff --end-of-options [base] -- [paths...]`                  | Inspect unstaged or merge-base diff          |
| `diffCached`       | `git diff --cached -- [paths...]`                                 | Inspect staged changes                       |
| `show`             | `git show --stat --patch --end-of-options <sha>`                  | Inspect a commit                             |
| `log`              | `git log --oneline -100`                                          | Commit picker / reviewer context             |
| `branches`         | `git branch --format=%(refname:short)`                            | Branch picker                                |
| `mergeBase`        | `git merge-base --end-of-options HEAD <branch>`                   | Base branch target resolution                |
| `revParseUpstream` | `git rev-parse --abbrev-ref --end-of-options <branch>@{upstream}` | Fallback base branch resolution              |

Implementation constraints:

- Use `spawn`/`execFile` with argument arrays, never shell string interpolation.
- Reject unsupported actions and arbitrary flags.
- Protect git ref arguments from option injection: `base`, `sha`, and `branch` must be non-empty, must not start with `-`, and must not contain NUL or control characters.
- Restrict `sha` to short or full commit hex (`^[0-9a-fA-F]{7,40}$`) before passing it to `git show`.
- Use `--end-of-options` for git commands that accept user-controlled refs, where supported, and keep pathspecs after `--`.
- Restrict optional paths to relative project paths; reject absolute paths, empty segments, `.` segments, `..` traversal, NUL/control characters, and path values beginning with `-`.
- Include regression tests for malicious values such as `--output=/tmp/pwned`, `-c`, path traversal, and embedded control characters.
- Truncate output using pi truncation utilities and report truncation.
- No write-capable git commands (`checkout`, `reset`, `stash`, `commit`, `apply`, `clean`, etc.).

### Review targets

Same 4 target types as the Rust implementation. The difference between them is the task prompt sent to the reviewer subprocess. The reviewer system prompt is identical across all 4.

| Target                  | Prompt                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| ----------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Uncommitted changes** | "Review the current code changes, including staged, unstaged, and untracked files. Start with the `reviewer-git` tool using `{ action: \"statusShort\" }`, inspect unstaged changes with `{ action: \"diff\" }`, inspect staged changes with `{ action: \"diffCached\" }`, and read relevant untracked files reported by status. Provide prioritized findings."                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| **Base branch**         | Preferred: "Review the code changes against the base branch '{{base_branch}}'. The merge base commit for this comparison is {{merge_base_sha}}. Use the `reviewer-git` tool with `{ action: \"diff\", base: \"{{merge_base_sha}}\" }` to inspect the changes relative to {{base_branch}}. Provide prioritized, actionable findings." Fallback: if direct merge-base resolution fails, the parent command validates the selected branch with the same ref guard used by `reviewer-git`, resolves `{{branch}}@{upstream}` with `pi.exec("git", ["rev-parse", "--abbrev-ref", "--end-of-options", "{{branch}}@{upstream}"])`, then resolves merge-base against that upstream. The fallback prompt is rendered only after the parent has a concrete `{{upstream_branch}}` and `{{merge_base_sha}}`; if either command fails, `/review` exits with a clear error instead of asking the reviewer to infer the base. |
| **Commit**              | "Review the code changes introduced by commit {{sha}} (\"{{title}}\"). Use the `reviewer-git` tool with `{ action: \"show\", sha: \"{{sha}}\" }` and read surrounding files as needed. Provide prioritized, actionable findings." With variant for missing title.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Custom**              | Verbatim user text. The system prompt still requires using available read-only tools and ending with `finish-review`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |

### Entry UX

- `/review` with no args and UI available → `ctx.ui.select()` with 4 options:
  1. Review against a base branch → branch picker via `ctx.ui.select()` using parent-process `pi.exec("git", ["branch", "--format=%(refname:short)"])`
  2. Review uncommitted changes → direct
  3. Review a commit → commit picker via `ctx.ui.select()` using parent-process `pi.exec("git", ["log", "--oneline", "-100"])`
  4. Custom review instructions → `ctx.ui.input()`
- `/review` with no args and no UI (`ctx.hasUI === false`) → review uncommitted changes.
- `/review <text>` → goes straight to Custom target in all modes.
- If branch/commit/custom input is cancelled in UI mode, the command exits without spawning a subprocess.

### Structured output

Register a companion extension loaded in the subprocess via `-e`. It registers two tools:

1. `reviewer-git` — restricted read-only git inspection.
2. `finish-review` — terminating structured-output tool.

`finish-review` uses the findings schema as its parameters. Pi validates arguments against the TypeBox schema. Its `execute()` implementation returns the submitted params in `details` and `terminate: true`:

```typescript
return {
  content: [{ type: 'text', text: 'Review submitted.' }],
  details: params,
  terminate: true,
}
```

Parser extraction order:

1. `tool_execution_start` events where `toolName === "finish-review"` and `args` match `ReviewOutput`.
2. assistant `message_end` events with `content` blocks of type `toolCall` and `name === "finish-review"`.
3. `tool_execution_end` / tool-result messages for `finish-review` where `result.details` or `message.details` match `ReviewOutput`.
4. Fallback: if `finish-review` was not called, do not attempt to parse assistant text as JSON. Wrap the last assistant text as a single unstructured review result in a complete default `ReviewOutput`:

```typescript
{
  findings: [],
  overall_correctness: "patch is correct",
  overall_explanation: rawText || "Reviewer produced no structured output.",
  overall_confidence_score: 0,
}
```

### Model selection

Fallback chain: `config.model ?? ${ctx.model.provider}/${ctx.model.id}`.

The user can pin a cheaper/faster model for reviews in config. If no config model exists, the reviewer uses the current session model. If neither exists, the command fails with a clear error before spawning the subprocess.

### Configuration

Single global YAML file at `${getAgentDir()}/review.yaml`. This respects `PI_CODING_AGENT_DIR`; use `getAgentDir()` consistently for the agent config directory instead of hardcoded home-directory paths.

System prompt lives as a separate file at `${getAgentDir()}/review-prompt.md` by default. Both files are created from embedded defaults on first run. The YAML is the source of truth after creation.

#### Config schema

```yaml
# Model for the reviewer (null = current session model)
model: null

# User-configurable reviewer subprocess tools.
# Include built-in tools and tools from configured extensions.
# pi-review's internal tools (reviewer-git, finish-review) are added automatically.
tools:
  - read
  - grep
  - find
  - ls

# Extra extensions to load explicitly with -e after --no-extensions.
# If an extension registers tools, list those tool names in tools above.
# TRUST: any extension runs with local process permissions and may weaken review isolation.
extensions: []

# Path to the reviewer system prompt. ~ and environment-derived agent dir are supported.
systemPrompt: ${getAgentDir()}/review-prompt.md

# Task prompts per review target.
# Template variables: {{base_branch}}, {{merge_base_sha}}, {{branch}}, {{upstream_branch}}, {{sha}}, {{title}}
prompts:
  uncommitted: 'Review the current code changes, including staged, unstaged, and untracked files. Start with the `reviewer-git` tool using `{ action: "statusShort" }`, inspect unstaged changes with `{ action: "diff" }`, inspect staged changes with `{ action: "diffCached" }`, and read relevant untracked files reported by status. Provide prioritized findings.'
  baseBranch: 'Review the code changes against the base branch ''{{base_branch}}''. The merge base commit for this comparison is {{merge_base_sha}}. Use the `reviewer-git` tool with `{ action: "diff", base: "{{merge_base_sha}}" }` to inspect the changes relative to {{base_branch}}. Provide prioritized, actionable findings.'
  baseBranchFallback: 'Review the code changes against ''{{branch}}'' via its upstream ''{{upstream_branch}}''. The merge base commit for this comparison is {{merge_base_sha}}. Use the `reviewer-git` tool with `{ action: "diff", base: "{{merge_base_sha}}" }` to inspect the changes relative to {{upstream_branch}}. Provide prioritized, actionable findings.'
  commit: 'Review the code changes introduced by commit {{sha}} ("{{title}}"). Use the `reviewer-git` tool with `{ action: "show", sha: "{{sha}}" }` and read surrounding files as needed. Provide prioritized, actionable findings.'
  commitNoTitle: 'Review the code changes introduced by commit {{sha}}. Use the `reviewer-git` tool with `{ action: "show", sha: "{{sha}}" }` and read surrounding files as needed. Provide prioritized, actionable findings.'

# Output filtering thresholds
thresholds:
  minConfidence: 0.0 # drop findings below this confidence_score
  maxPriority: 3 # drop findings above this priority number (P0=0, P3=3); missing priority is treated as P3 for filtering/display
```

Validation rules:

- `tools` is the user-configurable subprocess tool allowlist and may include built-in tools and configured extension tools.
- Every entry in `tools` must match pi tool-name syntax.
- `reviewer-git` and `finish-review` are reserved internal tool names. If users include them in `tools`, config loading fails with a message explaining they are added automatically.
- The final subprocess `--tools` value is `config.tools` plus internal `reviewer-git,finish-review`.
- Invalid thresholds fail config loading with a clear message.
- Findings with missing `priority` are treated as P3 (`priority = 3`) for threshold filtering and display, while preserving the original structured finding details unchanged.
- `extensions` must be an array of strings.
- `bash`, `write`, `edit`, configured extension paths, and unknown/non-default tool names are allowed but produce a startup warning because they may weaken the default read-only review isolation.

### Result injection

The parent extension injects a custom message:

```typescript
pi.sendMessage({
  customType: 'review',
  content: formatReviewForContext(reviewOutput),
  display: true,
  details: reviewOutput,
})
```

Rules:

- `content` contains the full formatted verdict, explanation, and findings because message `details` are renderer metadata and are not the primary LLM-readable content.
- `details` stores the original structured `ReviewOutput` for TUI rendering and future programmatic use.
- `pi.registerMessageRenderer("review", ...)` renders the same findings compactly in the TUI, with expanded details available.

### Progress and cancellation

- **Progress**: `ctx.ui.setStatus("review", "Reviewing...")` once at start, cleared on completion.
- **Cancellation**:
  - Store the child process reference at extension scope.
  - Register `session_shutdown` to terminate any active subprocess.
  - Wire command-level cancellation to the child process where an `AbortSignal` is available.
  - Kill with `SIGTERM`, then `SIGKILL` after a short grace period if still running.
- **Cleanup**: Prefer `AsyncDisposable` / `await using` guards for status and child process lifecycle where the pi extension API supports it. Fall back to `try/finally` where it doesn't.

### Command handler orchestration

The `/review` command handler is the top-level orchestrator. It runs this sequence:

1. Load config.
2. Target selection (picker, no-UI default, or inline args → custom).
3. Resolve task prompt (template rendering, base-branch merge-base resolution).
4. Read system prompt from config path.
5. Set status (`ctx.ui.setStatus`).
6. Spawn reviewer subprocess, wait for completion.
7. Parse output (extract findings or fallback).
8. Apply threshold filtering.
9. Inject result via `pi.sendMessage({ customType: "review", ... })`.
10. Cleanup via `AsyncDisposable` guards where the API supports it, otherwise `try/finally`.

Every step below the handler (prompt resolution, subprocess spawning, output parsing, rendering) is a dependency injected into the handler. The handler defines the interfaces these dependencies must satisfy.

### Reviewer system prompt

Port the Rust implementation's `review_prompt.md` with one modification: replace the "output strict JSON" section with instructions to call the `finish-review` tool as the final action.

Retained from the Rust prompt:

- Role: "acting as a reviewer for a proposed code change made by another engineer"
- Bug-flag guidelines (8 rules)
- Comment-quality rules (8 rules)
- Quantity rule
- Style rules
- Priority tagging: P0 (drop everything) through P3 (nice-to-have)
- Verdict: `overall_correctness` of `"patch is correct"` or `"patch is incorrect"`

Modified:

- Output format: call `finish-review` with findings instead of emitting strict JSON.
- The reviewer must not suggest changes unless they are actionable review findings.
- The reviewer must use `reviewer-git` for diffs and must not ask for `bash`.
- If there are no qualifying findings, call `finish-review` with an empty `findings` array.

### `finish-review` tool schema

```typescript
Type.Object({
  findings: Type.Array(
    Type.Object({
      title: Type.String(),
      body: Type.String(),
      confidence_score: Type.Number({ minimum: 0, maximum: 1 }),
      priority: Type.Optional(Type.Integer({ minimum: 0, maximum: 3 })),
      code_location: Type.Object({
        absolute_file_path: Type.String(),
        line_range: Type.Object({
          start: Type.Integer({ minimum: 1 }),
          end: Type.Integer({ minimum: 1 }),
        }),
      }),
    }),
    { maxItems: 50 }
  ),
  overall_correctness: StringEnum([
    'patch is correct',
    'patch is incorrect',
  ] as const),
  overall_explanation: Type.String(),
  overall_confidence_score: Type.Number({ minimum: 0, maximum: 1 }),
})
```

`Type` comes from `typebox`, which pi uses for extension tool parameter schemas. Declare `typebox` as a `peerDependency` rather than bundling it, per pi package guidance.

`StringEnum` comes from `@earendil-works/pi-ai` for provider compatibility.

## Implementation requirements

- Never use sync filesystem APIs (`existsSync`, `readFileSync`, `writeFileSync`, etc.). Use `fs/promises`.
- Build subprocess command arguments in a pure function and test them.
- Subprocess args must use `--system-prompt` and must include `--no-context-files`, `--no-skills`, `--no-prompt-templates`, and `--no-extensions` by default.
- Use `spawn`/`execFile` with argument arrays for `reviewer-git`; never shell strings.
- Do not pass secrets, environment dumps, or config contents into review output.
- Treat config `extensions` and `tools` as explicit isolation overrides in README and config comments.

## Tests

Tests only for pure logic and controlled subprocess-contract helpers, using vitest:

- `config/schema.test.ts` — generated default config is valid, `tools` is treated as the source of truth for user-configurable tools, reserved internal tool names (`reviewer-git`, `finish-review`) in config fail validation, invalid tool-name syntax fails validation, invalid thresholds fail validation, invalid escape-hatch shapes fail validation.
- `prompts/task-prompts.test.ts` — template rendering, uncommitted prompt includes staged/unstaged/untracked instructions, base branch direct prompt, base branch fallback prompt with concrete `upstream_branch` and `merge_base_sha`, and no pseudo-syntax placeholders.
- `features/subprocess/args.test.ts` — command args include `--system-prompt`, `--no-context-files`, `--no-skills`, `--no-prompt-templates`, `--no-extensions`, explicit `-e`, final `--tools` equal to `config.tools` plus internal `reviewer-git,finish-review`, no `--append-system-prompt`, no `--dangerously-skip-permissions`, no `bash` in the generated default config, the companion extension is loaded before configured extensions, and configured extension tool names come from `tools`.
- `features/subprocess/reviewer-git.test.ts` — allowed action to git argv mapping; rejects unsupported actions, absolute paths, path traversal, write-capable commands, refs that start with `-`, refs containing NUL/control characters, non-hex `sha` values, paths that start with `-`, and malicious values such as `--output=/tmp/pwned`.
- `features/subprocess/parse-output.test.ts` — extracting findings from `tool_execution_start`, assistant tool-call content, tool result details, threshold filtering, missing-priority-as-P3 filtering/display behavior, and complete fallback output.
- `features/rendering/format-findings.test.ts` — context-format output includes full findings, verdict, explanation, priorities, and code locations.

No tests for full pi runtime wiring (actual command registration, message renderer registration, live subprocess model calls). These involve the full pi runtime and are covered by manual smoke testing.

## Implementation phases

Top-down: start from the user-facing entry point and work down to underlying implementation. Each phase defines the interfaces it needs from the layers below before those layers exist.

### Phase 1 ✅ — package skeleton, config, and subprocess contract

Package skeleton, config schema/load/paths/defaults, subprocess arg builder.

### Phase 2 — types, command handler, target picker, root wiring

Types (`ReviewTarget`, `ReviewOutput`/`Finding`), command handler skeleton, target picker, root `index.ts` wiring (command registration, message renderer stub, `session_shutdown` handler).

### Phase 3 — prompt resolution

Task prompt template rendering, system prompt reader, ref validation, base-branch merge-base resolution.

### Phase 4 — rendering and result injection

`formatReviewForContext`, `formatReviewForDisplay`, message renderer.

### Phase 5 — subprocess spawn and output parsing

Subprocess spawning (JSON capture, cancellation), output parser (extract `finish-review` findings, fallback, threshold filtering).

### Phase 6 — companion extension tools and reviewer prompt

`reviewer-git` (action→argv mapping, ref/path guards), `finish-review` tool, companion extension registration, `review-prompt.md`.

### Phase 7 — smoke testing and README

Manual testing, documentation.
